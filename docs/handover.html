How a reservation enquiry travels from the Squarespace form, through Monday.com and the attribution stack, to a server-side Step 4 booking conversion uploaded to Google Ads.

1. The pipeline at a glance

Student Luxe is a curating agent, not an operator. It earns commission per booking. Attribution is fully in-house: no third-party tool. Every enquiry is captured, deduped, written to Monday, emailed, and uploaded to Google Ads as a conversion. Bookings that later confirm upload again at their real revenue.

Two supporting daily jobs keep the data clean

enrich-attribution (06:00 UTC): back-fills a lead's campaign / ad group / keyword from its gclid via Google's click_view, because UTMs sitting in a tracking template never reach the landing page under parallel tracking.

sync-booking-values (06:30 UTC): recomputes the true booking commission (a Monday formula the API cannot read) and writes it to the number column Google reads, so new bookings fill in and amended ones re-sync.

2. Stage 1: capture on the Squarespace form

The reservation form lives in a single Squarespace Code Block (repo luxe-enquiry-form). A tracking script writes ad-click identifiers and UTM values into sl_ cookies as the visitor lands, then posts them with the enquiry.

What gets captured

GroupFields
Google click IDsgclid, gbraid (iOS web), wbraid (iOS app). Auto-tagged by Google on every ad click.
UTM / campaignutm_campaign, utm_adgroup, utm_term, utm_matchtype, utm_content.
First-touch twinssl_first_* written once, never overwritten. Preserves the original source across return visits.
Session + journeysl_session_id, landing page, last page, visited paths (page-view pings via /api/track).
Socialfbclid. Defaults to channel "Instagram" (auto-injected on any IG outbound link, not a paid signal).

Critical gotcha: UTMs in a tracking template never reach the page

Google Ads uses parallel tracking (mandatory on Search). The browser goes straight to the Final URL; the tracking template fires only as a background beacon. So any utm_* that lives in the account/campaign tracking template never lands in the address bar, and the form never sees it. Only the auto-tagged gclid/gbraid arrive. On top of that, the campaign NAME has no valid ValueTrack macro ({campaign} resolves empty, {adgroupname} lands as a literal), which is why campaign detail has been missing on most PPC leads.

Handled by: /api/enrich-attribution looks each lead's gclid up in Google's click_view and writes the real campaign / ad group / keyword / match type back to Monday daily. Recoverable for ~90 days (click_view retention) and only for Search/Shopping clicks.

3. Stage 2: /api/submit-enquiry

The main handler. Runs in strict order, never skipping a step. The Stay Luxe sister form (/api/submit-stayluxe) mirrors this exactly, minus the Google Ads upload (Stay Luxe does not run ads).

  1. IP blocklist rejects known spam IPs silently (returns a fake 200 so bots learn nothing).
  2. Attribution capture (buildTouch): reads click IDs, UTMs, geo, device, browser from cookies plus headers, and loads any existing KV session by sl_session_id.
  3. Duplicate detection (see below), runs before every Monday write.
  4. Monday Leads write to board 2171015719, with retry on transient API blocks.
  5. Lead source + channel classification (computeLeadSource): msclkid > Bing organic > gclid/utm (Google Ads) > fbclid (Instagram) > direct > Google organic.
  6. Resend emails to the guest and the team, fired in parallel.
  7. Google Ads Step 1 upload via the Data Manager API (see section 5).
  8. KV session attach plus email/Monday lookup indices for later joining.

Duplicate detection: 4 signals

Each candidate lead is scored on email, phone, IP, and name. 2 or more matches flags the Monday row as a Possible Duplicate and copies the original assignee.

Same-household exception

An IP-only match (1 of 4) also flags when the original lead is under 14 days old. This catches a parent and student enquiring separately from the same home wifi: unique names, emails, and phones, but one IP. Older IP-only matches stay unflagged because shared IPs (halls, offices, mobile CGNAT) are too noisy alone.

When a Monday write fails

The guest still gets their confirmation. A rescue email to the team carries the full tracking payload (campaign, ad group, term, all click IDs, session id, landing page, first-touch fields, visited paths, IP) so the lead can be added by hand with zero attribution loss.

4. The Google Ads conversion funnel

Four conversion actions, one per stage of intent. All set to Count = Every (gbraid/wbraid require it). Steps 2 to 4 use Enhanced Conversions for Leads: they match on hashed email plus phone, so no live click ID is needed even months later.

StepActionFires whenValueMatch method
1NEW (server-side enquiry)Form submitted£1Click-based (gclid/gbraid/wbraid)
2Moderate PotentialLead tagged Moderate in Monday£150EC for Leads (email+phone)
3High PotentialLead tagged High in Monday£300EC for Leads
4Confirmed BookingBooking confirmed / revenue filledReal commissionEC for Leads

Step 2 and 3: lead qualification

/api/submit-high-potential is a Monday webhook. When a lead's potential status changes, it pulls the linked lead's email, phone, and name, and uploads Step 2 (£150) or Step 3 (£300). The event timestamp is stamped at webhook time, not lead-creation time, so an old lead does not trip Google's acceptable-time window.

Benched: Step 3.5 "Qualified Leads" / pending bookings

Open idea: upload the moment a guest signals intent (pending / deposit), before confirmation. Decision: do not put pending into Step 4, it pollutes value-based bidding because some pending bookings cancel. If built, it must be its own action ("Booking Requested") at a discounted value (estimated commission × measured close rate), Secondary/observe-only first. Blockers: RETRACT/RESTATE lives on the deprecated Google Ads API we left; the adjustment window is only 55 days; and Step 4 volume may already be enough. Parked.

5. Stage 4: the confirmed booking upload

/api/submit-booking is a Monday webhook on the Bookings board. It fires on two triggers: the booking status changing (any status counts as confirmed except "Pending Booking", a lost or cancelled booking can still carry a retained fee), or the revenue column numeric_mm1ge9h4 being filled. Only PPC bookings upload, and only once a value is present. The stable transaction id dedupes the repeat fires across a booking's lifecycle, so Google never double-counts.

Sequence

  1. Fetch the booking plus its linked lead (email, phone, first/last name, click IDs) from Monday.
  2. Confirm it is PPC and has a positive value.
  3. Upload Step 4 via Data Manager, value = the commission, event timestamp = now.
  4. Transaction id = booking.created_at + email (stable), so a later re-upload of the same booking is deduped, not double-counted.
  5. Log to the KV event journal; email a success or failure alert.

The booking value problem

Monday cannot serve the real value over the API

The true commission is a Monday formula column (formula2, "Total Luxe Commission excl. VAT"). It references cross-board mirror columns, and Monday never evaluates a mirror-referencing formula server-side, so the API returns null. It can only be read by a human in the UI. That is why the value has had to be hand-copied into the plain number column numeric_mm1ge9h4 ("Rev to Google") that the upload actually reads.

Handled by /api/sync-booking-values (daily)

The formula is reverse-engineered and recomputed in code from its 10 base columns (nightly rate, nights, commission %, VAT/Net-Gross mirrors, discount). It writes the result to numeric_mm1ge9h4 when it differs by more than £1: blank bookings fill in, amended bookings (guest changes nights) re-sync. Matches the hand-entered value on ~90% of PPC bookings; the rest are extensions/rebookings (instalment-column logic not modelled) or genuinely stale entries, all listed in a daily email digest. Monday's formula string comparison is case-sensitive, replicated exactly.

Side effect worth knowing

Editing numeric_mm1ge9h4 (by hand or by the sync job) re-fires the submit-booking webhook, which re-uploads Step 4 with the new value under the same transaction id. Intended: it keeps Google in sync with amendments. Cost: a burst of re-uploads can trigger transient upload failures (see next page).

6. The Data Manager API layer

All uploads go through api/_dataManager.js. This replaced the deprecated googleads.googleapis.com/.../uploadClickConversions path, which now returns ALLOWLIST errors.

Request shape

PartDetail
EndpointPOST https://datamanager.googleapis.com/v1/events:ingest
AuthAuthorization: Bearer + x-goog-user-project: <GCP_PROJECT_ID> (required). Scope datamanager.
Dropped headersdeveloper-token and login-customer-id are GONE. MCC mapping moved into the body (loginAccount.accountId).
Event fieldsdestinationReferences, transactionId, eventTimestamp (RFC3339), eventSource: "WEB", optional adIdentifiers, userData.userIdentifiers (hashed email/phone), currency, conversionValue.

Three failure modes we hit and fixed

Transient generic 400. Data Manager intermittently rejects a valid payload with a bare events.events[N] violation and no named field. Detected and retried (with 5xx/429) using exponential backoff. Anything that names a real sub-field is a deterministic bug and is not retried.

EVENT_TIME_INVALID. Caused by trusting a client-supplied timestamp that could be stale or clock-skewed. Every handler now stamps the event at server time.

Address identifier 400. The address block needs first name + last name + region + postcode all present or the whole request fails. We do not collect postcode, so the address block is omitted; matching relies on email + phone.

Recovery: /api/replay-failed-events (runs every 4h)

Scans the KV gads:events fail log, re-fetches each lead/booking from Monday (values pulled fresh, source of truth), and re-uploads via Enhanced Conversions for Leads. Idempotent per Monday item, so re-running never double-counts. This is also the alerting layer: it emails only the failures that stay broken past 6 hours (see Monitoring).

Failure alerts: only when it does not self-heal

Google's Data Manager API intermittently rejects valid uploads with a transient 400 (~1 in 5). Alerting on every one would be pure noise, because most recover on retry. So the alert model is two-layered:

  • Every upload attempt is logged to the KV gads:events sorted set (35-day retention). A failure logs but does not email.
  • The replay cron (every 4 hours) re-uploads anything still failing. It skips fails that already recovered (a later success exists for the same booking/lead).
  • Only when a failure is still unrecovered after 6 hours does it email alex@studentluxe.co.uk, once per item (deduped), with the exact failure reason.

What triggers an email now

A Step 1 / 2 / 3 / 4 upload that has genuinely stayed broken for 6+ hours: a persistent API error, or a PPC booking/lead with no email or phone on the Monday row (nothing to match on, so it can never upload). Transient blips that the replay fixes stay silent. The daily digest still summarises everything, healed or not.

Ignoring an unfixable upload

Some conversions can never upload, e.g. a lead mislabelled PPC with no email, phone, or gclid. Left alone they would re-alert forever. /api/gads-ignore manages a small ignore list so the replay skips them entirely:

  • Ignore: /api/gads-ignore?secret=…&mondayId=123&action=Confirmed%20Booking
  • Undo: add &remove=1  ·  List: call with no mondayId

Note: ignoring only silences the Google upload. If a lead is mislabelled PPC, also correct its source in Monday so it stops skewing PPC revenue reporting.

Application errors (not just Google Ads)

Vercel keeps runtime logs for only ~1 hour, so a daily dashboard scan cannot see 24 hours of errors. Instead every handler's top-level catch writes to a Redis app:errors log (14-day retention). /api/error-digest runs daily at 07:00 UTC, groups the last 24h by endpoint and message, and emails a summary. No email is sent on a zero-error day. This covers any unexpected failure across the whole project (Monday writes, email sends, webhooks, crons), separate from the Google Ads upload alerts. Pure platform errors (function timeout, out-of-memory) do not reach our code and would need a Vercel Log Drain to capture.
